# 项目通用规范

## 1. 适用范围

本文档提取当前互联网医院项目的通用协作、编码、数据库、接口、前端、脚本和交付规范，适用于后续新增功能、问题修复、模块重构、文档维护和提交评审。

如本文档与仓库根目录 `AGENTS.md` 冲突，以 `AGENTS.md` 为准。

## 2. 协作红线

1. 文档、注释、提交信息、PR 描述一律使用中文。
2. 代码标识符保持英文，包括类名、方法名、变量名、包名、接口路径和数据库字段名。
3. 不主动执行编译、测试、构建命令，包括但不限于 `mvn compile`、`mvn test`、`pnpm test`、`pnpm build`。
4. 不回滚用户已有改动，不使用破坏性 Git 命令。
5. 修改范围必须贴合当前任务，避免顺手重构无关模块。
6. 新增功能、接口、页面、数据库表或环境变量后，必须同步更新文档和接口脚本。

## 4. 后端分层规范

后端模块统一采用以下包结构：

```text
com.hlw.<module>
├── controller
├── service
├── service.converter
├── mapper
├── entity
├── domain.req
├── domain.resp
├── config
└── client
```

分层职责：

1. Controller 只负责接收参数、参数校验、入口日志、调用 Service、返回 `R`。
2. Service 负责业务编排、事务、状态流转、租户校验、异常处理和 VO 转换。
3. Mapper 统一继承 MyBatis-Plus `BaseMapper`，不在 Controller 中直连数据库。
4. Entity 只表达持久化结构，不承载复杂业务流程。
5. Req/Resp 只表达接口入参与出参，不混入数据库读写逻辑。
6. Converter 负责 Entity、Req、Resp 之间的转换，避免转换逻辑散落。
7. Feign Client 只声明跨服务接口，不写业务判断。

## 5. Java 编码规范

1. 每个类必须有中文 Javadoc，说明职责。
2. 每个 public 方法必须有中文 Javadoc，包含描述、`@param`、`@return`；无返回值方法可省略 `@return`。
3. 关键 private 方法也应添加中文 Javadoc，尤其是状态判断、租户校验、跨表查询、转换和兜底逻辑。
4. Entity、DTO、VO、Req、Resp 每个字段必须有中文行注释。
5. Controller 入口方法和关键 Service 方法必须记录 `log.info`；异常、兜底、跳过和关键分支记录 `log.warn`。
6. 日志禁止输出密码、token、验证码、身份证号、手机号全量、密钥等敏感信息。
7. 写操作使用 `@Transactional(rollbackFor = Exception.class)`。
8. 只读查询建议使用 `@Transactional(readOnly = true)`。
9. 业务异常统一使用 `BizException` 或已有公共异常能力。
10. Lombok 使用遵守 `01-Lombok使用规范.md`，Entity 默认使用 `@Getter`、`@Setter`，禁止无脑使用 `@Data`。

## 6. 接口规范

接口路径按服务域划分：

- `/auth/**`
- `/gateway/**`
- `/system/**`
- `/patient/**`
- `/doctor/**`
- `/consult/**`
- `/appointment/**`
- `/prescription/**`
- `/drug/**`
- `/order/**`

REST 约定：

| 操作 | 方法 | 示例 |
| --- | --- | --- |
| 列表查询 | `GET` | `/system/user` |
| 详情查询 | `GET` | `/system/user/{id}` |
| 创建 | `POST` | `/system/user` |
| 更新 | `PUT` | `/system/user/{id}` |
| 删除 | `DELETE` | `/system/user/{id}` |
| 业务动作 | `POST` | `/order/orders/{id}/pay` |

响应约定：

1. Controller 返回值统一使用 `R<T>`。
2. 分页接口统一使用 `PageQuery` 入参和 `PageResult<T>` 出参。
3. 新增接口后必须在 `resources/scripts/api-test.sh` 的 `run_all_cases()` 中补充用例。
4. 接口测试用例应包含正常场景和关键参数，能串联创建、详情、更新、删除时优先串联。

## 7. 数据库规范

1. 当前 MySQL 8 基线脚本从 `resources/sql/001-mysql8-baseline.sql` 开始维护。
2. 新增 schema 变更优先新增顺序 SQL 文件，例如 `002-xxx.sql`、`003-xxx.sql`。
3. 每张表必须有中文表注释。
4. 每个字段必须有中文字段注释。
5. 逻辑删除字段统一使用 `deleted`，按当前模块约定维护 `0` 未删除、`1` 已删除。
6. 多租户业务表必须明确 `tenant_id` 设计，平台级表或关系表是否忽略租户过滤必须在代码中显式处理。
7. 常规查询必须考虑租户隔离和逻辑删除条件。
8. 系统用户业务编号统一使用 `user_id VARCHAR(34)`，格式为 `U_` 加 32 位 UUID。
9. 后台按钮权限统一走 `sys_menu.perms`，不再维护独立权限码模块。

## 8. 认证与租户规范

1. 网关负责清理外部传入的租户头，并根据登录令牌重新写入可信 `X-Tenant-Id`。
2. 业务服务通过 `common-security` 的租户上下文过滤器读取可信租户和登录主体。
3. JWT 中必须包含 `userId`、`tenantId`、`userType`。
4. JWT 解析入口优先使用 `common-core` 中的统一解析能力，避免各模块重复解析 claims。
5. 登录接口优先使用网关透传的可信租户头，缺少时读取请求体租户编号。
6. 平台租户编号为 `0`，隔离租户编号为 `-1`。
7. 平台级数据访问必须显式说明是否忽略租户过滤，不能隐式绕过租户隔离。

## 9. 前端规范

1. 前端样式必须收口到 `.css` 文件，组件中通过 `className` 引用。
2. 禁止新增内联 `style={{ ... }}` 和 CSS-in-JS。
3. 管理端页面优先复用现有 `ModulePage`、统一 API 客户端和全局样式。
4. API 请求统一携带认证头和租户上下文，避免页面各自手写协议。
5. 新增页面或页面能力必须更新 `code/frontend/README.md`。
6. 页面文本、按钮、表单项和提示语使用中文。

## 10. 文档与脚本规范

新增功能必须同步更新：

1. 后端新增模块、接口、数据库表、环境变量：更新 `code/backend/README.md`。
2. 前端新增页面、路由、应用能力：更新 `code/frontend/README.md`。
3. 新增接口：更新 `resources/scripts/api-test.sh`。
4. 新增规范：更新 `resources/开发规范/README.md`。
5. 新增数据库结构：更新 SQL 脚本，并补齐中文注释。

接口脚本要求：

1. 默认通过 `POST /auth/login` 获取 token。
2. 后续请求统一带 `Authorization: Bearer <token>` 和 `X-Tenant-Id`。
3. 可以串联的 CRUD 用例应提取返回 `data.id` 或 `data.key` 继续执行详情、更新、删除。
4. 不适合 curl 自动断言的场景可以记录跳过原因。

## 11. Git 规范

1. 提交信息必须使用中文。
2. 提交前必须检查工作区状态，确认没有误提交无关文件。
3. 提交前至少执行非编译类静态检查，例如 `git diff --check`、`rg` 残留扫描。
4. 不主动执行编译测试命令；如确需运行，必须由用户明确允许。
5. 一次提交应聚焦同一类任务，避免把无关改动混在一起。

- [ ] 是否新增功能，并已更新对应 README。
- [ ] 是否新增或修改数据库表字段，并已补齐中文注释。
- [ ] 是否新增 public 方法，并已补齐中文 Javadoc。
- [ ] 是否新增 Entity/DTO/VO 字段，并已补齐中文行注释。
- [ ] 是否存在敏感信息日志输出。
- [ ] 是否误改无关文件。
- [ ] 是否执行 `git diff --check` 等静态检查。
