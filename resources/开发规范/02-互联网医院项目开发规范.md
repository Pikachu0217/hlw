# 互联网医院项目开发规范

## 1. 目的

本文档基于当前仓库结构、后端模块、前端应用、数据库脚本、接口测试脚本和现有 README 扫描整理，用于后续新增功能、修复缺陷、扩展模块和维护脚本时统一执行。

本规范与仓库根目录 `AGENTS.md` 保持一致；如发生冲突，以 `AGENTS.md` 的项目工作规则为准。

## 2. 技术栈与项目边界

当前项目为互联网医院多模块应用：

- 后端目录：`code/backend/`
- 前端目录：`code/frontend/`
- 数据库基线脚本：`resources/sql/init.sql`
- 接口测试脚本：`resources/scripts/api-test.sh`
- 一键启停脚本：`resources/scripts/service.sh`
- 项目规范目录：`resources/开发规范/`

后端技术栈：

- Java 17
- Spring Boot 3.2.12
- Spring Cloud 2023.0.6
- Spring Cloud Alibaba 2023.0.3.4
- MyBatis-Plus 3.5.16
- PostgreSQL 16
- Redis 7
- Nacos 2.3
- RabbitMQ
- Lombok

前端技术栈：

- React 18
- TypeScript
- Vite
- pnpm workspace
- `admin-web` 使用 Ant Design
- `patient-h5` 使用 Ant Design Mobile 和 Zustand

## 3. 通用协作规范

1. 文档、注释、提交信息和 PR 描述一律使用中文。
2. 类名、方法名、变量名、包名、接口路径等代码标识符保持英文。
3. 不主动执行编译、测试或打包命令，包括但不限于 `mvn compile`、`mvn test`、`pnpm test`、`pnpm build`。
4. 新增功能必须同步更新对应 README。
5. 新增后端接口必须同步更新 `resources/scripts/api-test.sh` 的 `run_all_cases()`。
6. 新增数据库表或字段必须同步更新 `resources/sql/init.sql`，并补齐表注释和字段注释。
7. 修改样式必须写入对应应用的 CSS 文件，组件内只通过 `className` 引用。
8. 不随意重构无关代码，不移动无关文件，不改写用户已有工作成果。

### 4.2 包结构规范

业务模块包名统一使用 `com.hlw.<module>`，典型结构如下：

```text
com.hlw.<module>
├── controller
├── service
├── service.converter
├── mapper
├── entity
├── domain.req / dto
├── domain.resp / vo
├── config
└── client
```

约束如下：

1. Controller 只负责接收参数、参数校验、记录入口日志、调用 Service、返回 `R`。
2. Service 负责业务编排、状态流转、事务、租户上下文校验、异常抛出和 VO 转换。
3. Mapper 统一继承 MyBatis-Plus `BaseMapper`，不在 Controller 或前端直连数据库。
4. Entity 只表达持久化结构，不承载复杂业务流程。
5. Converter 负责 Entity、DTO、VO 之间的转换，避免转换逻辑散落在 Controller。
6. Feign Client 只放跨服务调用声明，不写业务判断。

### 4.3 Controller 规范

1. 类必须加中文 Javadoc，说明控制器职责。
2. public 接口方法必须加中文 Javadoc，包含描述、`@param`、`@return`。
3. 类建议使用 `@RestController`、`@RequestMapping`、`@RequiredArgsConstructor`、`@Slf4j`。
4. 入参对象必须使用 `@Valid @RequestBody`，路径参数使用 `@PathVariable`。
5. 返回值统一使用 `R<T>`。
6. 每个入口方法必须记录 `log.info`，日志包含接口动作和关键入参。
7. 日志不得输出密码、token、身份证号、手机号全量、验证码等敏感信息。

示例：

```java
/**
 * 后台用户管理控制器。
 */
@RestController
@RequestMapping("/system/user")
@RequiredArgsConstructor
@Slf4j
public class UserController {
    /** 后台用户聚合服务。 */
    private final UserService userService;

    /**
     * 查询后台用户详情。
     *
     * @param id 用户编号
     * @return 后台用户详情
     */
    @GetMapping("/{id}")
    public R<UserResp> detail(@PathVariable Long id) {
        log.info("查询后台用户详情，id={}", id);
        return R.ok(userService.getUser(id));
    }
}
```

### 4.4 Service 规范

1. 类必须加中文 Javadoc，说明业务职责。
2. public 方法必须加中文 Javadoc，包含描述、`@param`、`@return`；无返回值方法可省略 `@return`。
3. 关键 private 方法也应补充中文 Javadoc，尤其是状态判断、租户校验、跨表查询和转换逻辑。
4. 写操作必须使用 `@Transactional(rollbackFor = Exception.class)`。
5. 只读查询建议使用 `@Transactional(readOnly = true)`。
6. 关键 Service 方法必须记录 `log.info`，异常、兜底、跳过和关键分支记录 `log.warn`。
7. 业务错误统一抛出 `BizException` 或使用现有公共异常能力。
8. 不在 Service 中返回裸 Entity 给 Controller，必须转换为 VO/Resp。

### 4.5 Entity、DTO、VO 规范

1. Entity、DTO、VO、Req、Resp 字段必须逐字段添加中文行注释。
2. Entity 继承 `BaseEntity` 时，公共字段由 `BaseEntity` 维护，不重复声明。
3. Entity 默认使用 `@Getter`、`@Setter`，禁止无脑使用 `@Data`。
4. DTO/Req 必须根据接口要求添加校验注解，如 `@NotBlank`、`@NotNull`、`@Min`。
5. VO/Resp 字段只放前端或调用方需要展示的数据，不暴露密码、token 摘要、内部配置等敏感字段。
6. Lombok 使用必须遵守 `01-Lombok使用规范.md`。

### 4.6 Mapper 与查询规范

1. Mapper 接口统一放在 `mapper` 包，命名为 `XxxMapper`。
2. 常规单表操作优先使用 MyBatis-Plus `BaseMapper`。
3. 查询条件优先使用 `LambdaQueryWrapper`，避免字符串字段名。
4. 查询业务数据必须考虑 `deleted` 逻辑删除和租户隔离。
5. 分页查询统一使用 `PageQuery` 和 `PageResult`。
6. 默认分页上限与后端分页插件保持一致，当前管理端拉取上限为 500。

### 4.7 事务、状态和异常规范

1. 多表写入、状态流转、扣减库存、锁定号源、支付回写必须放在事务中。
2. 状态值应优先沉淀为枚举或常量，不在业务代码中散落魔法字符串。
3. 删除默认使用逻辑删除，字段为 `deleted`。
4. 未找到数据统一返回业务异常，不返回空对象伪装成功。
5. 模块内可保留 `*ExceptionHandler`，但返回结构必须仍是 `R`。
6. 参数校验失败返回业务码 `400`，未知异常返回业务码 `500` 并记录 `log.error`。

### 4.8 认证与租户规范

1. 管理端登录前通过 `GET /system/tenant/options` 获取可登录租户的最小选项字段。
2. 登录接口 `POST /auth/login` 返回 JWT，JWT 包含用户、租户和用户类型信息。
3. 网关负责移除外部伪造的租户头，并根据登录令牌重新写入可信 `X-Tenant-Id`。
4. 业务服务通过 `common-security` 的租户上下文过滤器获取可信租户。
5. 写操作必须校验有效业务租户上下文，不能默认落到平台租户。
6. 平台级数据必须显式说明租户编号和访问条件。

## 5. 接口设计规范

### 5.1 路径规范

接口路径按服务域划分：

- `/auth/**`
- `/gateway/**`
- `/system/**`
- `/patient/**`
- `/doctor/**`
- `/consult/**`
- `/appointment/**`
- `/prescription/**`
- `/drug/**`
- `/order/**`

REST 风格约定：

| 操作   | 方法       | 示例                       |
|------|----------|--------------------------|
| 列表查询 | `GET`    | `/system/user`           |
| 详情查询 | `GET`    | `/system/user/{id}`      |
| 创建   | `POST`   | `/system/user`           |
| 更新   | `PUT`    | `/system/user/{id}`      |
| 删除   | `DELETE` | `/system/user/{id}`      |
| 业务动作 | `POST`   | `/order/orders/{id}/pay` |

### 5.2 响应规范

后端响应统一使用：

```java
public record R<T>(int code, String message, T data) {}
```

成功响应：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

分页响应统一使用 `PageResult<T>`，字段保持与前端请求层兼容：

- `records`
- `total`
- `pageNum`
- `pageSize`

### 5.3 新增接口联动规范

新增任何 Controller 接口后必须同步完成：

1. 更新 `resources/scripts/api-test.sh` 的 `run_all_cases()`。
2. 测试用例至少覆盖正常请求和关键参数。
3. 如接口用于管理端，更新 `code/frontend/admin-web/src/api/modules.ts` 或对应 API 文件。
4. 如接口用于患者端，更新 `code/frontend/patient-h5/src/app/api.ts` 或对应 API 文件。
5. 更新 `code/backend/README.md` 的接口说明。
6. 如果新增页面或前端流程，更新 `code/frontend/README.md`。

## 6. 数据库规范

### 6.1 基线脚本

当前数据库基线统一维护在 `resources/sql/init.sql`，不要在其他位置新增分散的临时建表脚本。

新增数据库对象时必须遵守：

1. 新增服务库使用 `CREATE DATABASE ... WHERE NOT EXISTS ... \gexec` 风格。
2. 新增表使用 `CREATE TABLE IF NOT EXISTS`。
3. 每张表必须有 `COMMENT ON TABLE`。
4. 每个字段必须有 `COMMENT ON COLUMN`。
5. 必要索引使用 `CREATE INDEX IF NOT EXISTS` 或 `CREATE UNIQUE INDEX IF NOT EXISTS`。
6. 演示数据插入必须可重复执行，优先使用 `ON CONFLICT DO NOTHING`。

### 6.2 表结构规范

业务表字段建议包含：

- `id`：主键编号。
- `tenant_id`：租户编号；平台级表可使用 `0`。
- 业务主字段：按领域命名，例如 `patient_name`、`doctor_id`。
- `create_time`：创建时间。
- `update_time`：更新时间。
- `create_by`：创建人编号。
- `update_by`：更新人编号。
- `deleted`：逻辑删除标识。

命名规则：

1. 表名使用小写下划线。
2. 系统模块表使用 `sys_` 前缀。
3. 认证模块表使用 `auth_` 前缀。
4. 网关模块表使用 `gw_` 前缀。
5. 预约模块表使用 `apt_` 前缀。
6. 问诊模块表使用 `con_` 前缀。
7. 处方模块表使用 `pre_` 前缀。
8. 订单模块表使用 `ord_` 前缀。
9. 患者模块表使用 `pat_` 前缀。
10. 医生模块表使用 `doc_` 前缀。
11. 药品模块表使用 `drug_` 前缀。

### 6.3 字段类型规范

1. 主键统一使用 `BIGSERIAL` 或与现有表保持一致的 `BIGINT`。
2. 金额字段应使用数值类型，不用字符串保存。
3. 状态字段可用 `VARCHAR` 或 `SMALLINT`，但同一业务域内必须统一。
4. 时间字段优先使用 `TIMESTAMP`，日期字段使用 `DATE`。
5. JSON 配置使用 `JSONB`。
6. 大文本使用 `TEXT`，普通名称和备注使用合理长度的 `VARCHAR`。

## 7. 前端开发规范

### 7.1 应用边界

前端工作区为 `code/frontend/`：

两个应用默认通过 Vite 代理访问网关：

```text
/api -> http://127.0.0.1:19000
```

### 7.2 目录规范

`admin-web` 常用目录：

```text
src/api
src/app
src/components
src/hooks
src/layouts
src/pages
src/router
src/store
src/styles
src/utils
```

`patient-h5` 常用目录：


新增页面时必须同步：

1. 增加页面组件。
2. 增加路由配置。
3. 管理端需要同步维护 `navigationTree`。
4. 如接入接口，增加 API 方法和类型。
5. 样式写入对应 `src/styles/global.css`。
6. 更新 `code/frontend/README.md` 的页面范围。

### 7.3 请求层规范

1. 管理端统一使用 `admin-web/src/api/client.ts` 中的 `apiClient`。
2. 患者端统一使用 `patient-h5/src/app/http.ts` 中的 `http`。
3. API 基础地址统一读取 `VITE_API_BASE_URL`，默认 `/api`。
4. 认证头统一使用 `Authorization: Bearer <token>`。
5. 管理端请求必须携带当前租户 `X-Tenant-Id`。
6. 请求和响应拦截器中保留必要 `console.info`、`console.warn`，方便联调排查。
7. 前端不拼接后端完整域名，除非明确是外部第三方服务。

### 7.4 组件与页面规范

1. 组件使用函数组件和 TypeScript。
2. 组件 Props 必须定义 interface 或 type。
3. 管理端列表页优先复用 `ModulePage`、`PageHero`、`MetricCard` 等现有组件。
4. 表格记录必须提供稳定 `key`。
5. 表单弹窗复用 `.module-form`、`.module-form__number` 等现有样式类。
6. 页面内不要写大段重复 UI，能复用现有组件时优先复用。
7. 页面文案使用中文，接口字段和类型名使用英文。

### 7.5 样式规范

1. 禁止在 JSX 中新增内联 `style={{}}`。
2. 禁止新增 CSS-in-JS。
3. 管理端样式统一维护在 `admin-web/src/styles/global.css`。
4. 患者端样式统一维护在 `patient-h5/src/styles/global.css`。
5. 组件只通过 `className` 引用样式。
6. 新增 class 命名应按页面或组件前缀组织，例如 `patient-console__header`。
7. 修改公共样式前必须评估所有引用页面，避免影响其他模块。

## 8. 脚本规范

### 8.1 接口测试脚本

接口测试脚本为 `resources/scripts/api-test.sh`。

维护要求：

1. 新增接口必须在 `run_all_cases()` 中增加用例。
2. 用例名称使用中文，能说明业务意图。
3. 请求体使用符合当前 DTO 的 JSON。
4. 创建后需要详情、更新、删除的接口，应尽量使用 `extract_data_id` 串联。
5. 不适合 curl 覆盖的接口使用 `record_skip_case` 明确记录跳过原因。
6. 新增服务路径时同步更新 `resolve_base_url()` 的直连端口映射。
7. 不在脚本中硬编码敏感密码，优先通过环境变量读取。

### 8.2 启停脚本

启停脚本为 `resources/scripts/service.sh`。

维护要求：

1. 新增后端模块时同步更新 `BACKEND_MODULES` 和 `MENU_BACKEND_MODULES`。
2. 新增前端应用时同步更新 `FRONTEND_APPS` 和 `MENU_FRONTEND_APPS`。
3. 新增命令必须有中文函数注释。
4. 脚本生成的运行时文件继续放在 `.runtime/pids` 和 `.runtime/logs`。
5. 脚本中涉及构建或打包命令时，文档必须说明代理默认不主动执行。

## 9. README 与文档规范

必须同步更新 README 的场景：

| 变更类型      | 必须更新                                                     |
|-----------|----------------------------------------------------------|
| 新增后端模块    | `code/backend/README.md`                                 |
| 新增后端接口    | `code/backend/README.md`、`resources/scripts/api-test.sh` |
| 新增数据库表或字段 | `code/backend/README.md`、`resources/sql/init.sql`        |
| 新增环境变量    | `code/backend/README.md` 或 `code/frontend/README.md`     |
| 新增前端应用    | `code/frontend/README.md`、`resources/scripts/service.sh` |
| 新增前端页面    | `code/frontend/README.md`                                |
| 新增前端接口接入  | `code/frontend/README.md`                                |

文档要求：

1. 全部使用中文。
2. 命令示例必须给出工作目录。
3. 如命令属于“仅供开发者自行执行”，必须明确代理不会主动执行。
4. 不写与当前实现不一致的承诺。
5. 文档和代码改动应放在同一次任务提交中。

## 10. 日志与安全规范

1. Controller、关键 Service、脚本关键函数必须有日志或中文注释。
2. 后端业务日志使用 `log.info`、`log.warn`、`log.error`。
3. 前端联调日志使用 `console.info`、`console.warn`。
4. 禁止输出密码、JWT 原文、验证码、身份证号、手机号全量、银行卡号。
5. 异常日志必须包含定位问题所需的业务主键或上下文。
6. 返回给前端的错误信息应可读，不暴露 SQL、堆栈、密钥和内部路径。

## 11. 交付检查清单

每次新增或修改功能完成前，按以下清单自查：

1. 是否遵守“不编译、不测试”的项目约束。
2. 是否所有新增文档、注释、提交说明均为中文。
3. 是否所有新增 public 方法都有中文 Javadoc。
4. 是否所有新增 Entity、DTO、VO、Req、Resp 字段都有中文注释。
5. 是否关键 Controller 和 Service 方法有日志输出。
6. 是否新增表和字段已写入 `resources/sql/init.sql` 且补齐注释。
7. 是否新增接口已更新 `resources/scripts/api-test.sh`。
8. 是否新增后端能力已更新 `code/backend/README.md`。
9. 是否新增前端页面或接口接入已更新 `code/frontend/README.md`。
10. 是否前端样式全部收口到 CSS 文件。
11. 是否未输出敏感信息。
12. 是否未引入与现有技术栈冲突的新依赖。

## 12. 后续维护原则

1. 本规范应随项目演进持续更新。
2. 新增专项规范时继续使用 `序号-中文名称.md` 文件名。
3. 已有专项规范优先沉淀稳定约束，避免在业务 README 中堆积过多实现细节。
4. 如果实际代码与规范不一致，应优先判断是历史债务还是规范过时，再决定修代码或修规范。
